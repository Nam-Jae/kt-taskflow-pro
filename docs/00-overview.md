## 읽는 순서

```
00-overview  →  01-product  →  02-specs  →  03-design  →  04-tasks  →  05-conventions
   (지도)          (왜)           (어떻게)      (어떻게 보이나)   (언제)          (어떻게 함께)
```

새 작업을 시작할 때마다 이 순서대로 읽는다.
특정 파일만 참조할 때도 반드시 `00-overview`를 먼저 읽어 전체 맥락을 확인한다.

## 왜 이렇게 나누는가 — 관심사 분리 원칙

하나의 거대한 문서는 "무엇을 고쳐야 하는지"를 찾기 어렵고, 팀원마다 필요한 정보가 달라 불필요한 부분을 매번 읽게 된다. 이 문서 구조는 **소프트웨어의 관심사 분리(Separation of Concerns)** 원칙을 문서에 그대로 적용한다.

| 관심사 | 질문 | 담당 파일 |
|--------|------|-----------|
| 제품 의도 | "무엇을, 왜 만드나?" | `01-product` |
| 기술 결정 | "어떤 기술로, 어떤 구조로 만드나?" | `02-specs` |
| 사용자 경험 | "화면이 어떻게 보이고 동작하나?" | `03-design` |
| 실행 계획 | "누가, 언제, 무엇을 하나?" | `04-tasks` |
| 협업 방식 | "어떻게 함께 작업하나?" | `05-conventions` |

각 파일은 **하나의 관심사만** 담는다. 덕분에:

- **수정 범위가 좁아진다** — 기술 스택이 바뀌면 `02-specs`만 열면 된다.
- **탐색 비용이 줄어든다** — "API 명세는 어디?" 라는 질문에 즉시 답할 수 있다.
- **충돌이 줄어든다** — 디자이너와 백엔드 개발자가 동시에 다른 파일을 수정해도 충돌이 없다.
- **AI 컨텍스트 효율이 높아진다** — 필요한 파일만 로드해 토큰을 낭비하지 않는다.
